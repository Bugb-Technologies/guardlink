# Handling Pentest Evidence

This document covers a category of files GuardLink writes during pentest workflows: **scan evidence containing live secrets**. If you run CXG templates against any real system, the resulting JSON files often contain JWTs, session cookies, Authorization headers, and credential payloads captured from successful exploits. This is the *point* of confirmation evidence — but it means those files need handling.

## What's in `.guardlink/pentest-findings/`

When `cxg scan` runs against a target and reports findings to GuardLink (via the closed-loop flow described in the README), the results are written to:

```
.guardlink/pentest-findings/
  └── <scan-id>.json     # one file per scan run
```

Each scan file is a structured record of every finding the scan produced. The shape includes an `evidence` block per finding:

```json
{
  "findings": [
    {
      "title": "SQL Injection → Admin JWT",
      "severity": "critical",
      "evidence": {
        "request": "POST /rest/user/login HTTP/1.1\nAuthorization: Bearer eyJ...",
        "response": "HTTP/1.1 200 OK\n{\"token\":\"eyJhbGc.eyJzdWIiOiJhZG1pbi...\"}",
        "data": { "extracted_role": "admin", "decoded_jwt": "..." }
      }
    }
  ]
}
```

That `evidence.response` may contain a fully decodable admin JWT. That's not a bug — it's the proof that the exploit worked. Without it, a pentest finding is unverifiable. But it does mean the file should be handled like any other artifact containing live credentials.

A related directory worth knowing about:

```
.guardlink/cxg-templates/
  └── *.py / *.yaml      # generated by `guardlink translate`
```

These templates often embed hard-coded test credentials, target URLs, and payload strings. Less sensitive than the findings JSON but still not always safe to commit to a public repo.

## Recommended ignore patterns

**Add the following to your repository's ignore file** before running any scans against systems you care about. GuardLink does not auto-write to your ignore file — that's deliberate, since not every project uses git, and not every team agrees on what should and shouldn't be checked in.

### Git (`.gitignore`)

```gitignore
# GuardLink — scan evidence and generated templates
.guardlink/pentest-findings/
.guardlink/cxg-templates/
```

### Mercurial (`.hgignore`)

```hgignore
syntax: glob
.guardlink/pentest-findings/**
.guardlink/cxg-templates/**
```

### Jujutsu

Jujutsu reads `.gitignore` by default in colocated repos; same content as the git block above. For non-colocated jj repos, use `jj file untrack` after the fact.

If your team uses a different VCS or ignore mechanism (Perforce p4ignore, Sapling, etc.), the same path prefixes apply — adapt the syntax to the tool.

## Operational guidance — sharing dashboards and reports externally

The ignore-file step handles **version-controlled** leakage. The other class of risk is **dynamic** — screenshots, exported HTML, demo screencasts, files emailed to vendors. The data still lives on disk and in memory. A few patterns to keep in mind:

- **Dashboard HTML output.** `guardlink dashboard . -o report.html` produces a standalone HTML file that embeds the full pentest data. If that file is shared, the embedded tokens go with it. Review before sharing.
- **Screenshots of the Pentest Findings tab.** Before screenshotting confirmed-finding evidence panels for a deck or blog post, check what's in the request/response strings. If the test target is a real production or staging system, assume live credentials are visible.
- **Markdown reports.** `guardlink report` does not currently embed full evidence into markdown output — it references the JSON files. Safe for sharing as long as the JSON files don't travel with it.
- **Recordings and demos.** A screencast capturing dashboard navigation may visually expose tokens for a fraction of a second. If you're recording for a public launch or pitch, run the scan against a test fixture (e.g. OWASP Juice Shop) where the credentials are well-known test values, not real secrets.

## Opt-in surgical redaction

For teams whose compliance posture requires that **no credential material exists in cleartext at rest** (regulated finance, healthcare, defense, some enterprise procurement policies), GuardLink supports a project-level configuration flag that applies surgical redaction at load time.

### Enabling

```bash
guardlink config set redact-evidence true
```

This writes `redactEvidence: true` to `.guardlink/config.json`. From the next dashboard / report / SARIF generation onward, all evidence is redacted as it loads — before any downstream consumer sees the data.

To disable:

```bash
guardlink config set redact-evidence false
```

Or to check current state:

```bash
guardlink config show
```

### What redaction preserves

The redaction is **surgical**, not blanket. The principle is: redact what enables replay, keep what proves the exploit. Specifically:

| Pattern | Treatment | Why |
|---|---|---|
| JWT `eyJhdr.eyJpld.sig` | Keep `eyJhdr.eyJpld.<signature-redacted>` | The decoded payload (`{"role":"admin"}`) is the *proof* of escalation. Anyone with a screenshot can decode the payload at jwt.io. Nobody can replay the token because the signature is gone. |
| `Authorization: Bearer <jwt>` | Same JWT split rule | Same logic. |
| `Authorization: Bearer <opaque>` | First 4 + last 4 chars: `gho_...ZZZZ` | Fingerprint for correlation, not enough for replay. |
| `Authorization: Basic dXNlcjp...` | `Authorization: Basic <redacted>` | Base64-decoded is `user:password`. No useful prefix to preserve. |
| `Authorization: Digest / NTLM` | Full value redacted | Same. |
| JSON: `"password":"value"` | `"password":"<redacted>"` | Field name preserved (proves the request used a password field — structural evidence). Value is the secret. |
| JSON: `"api_key"`, `"access_token"`, `"refresh_token"`, `"secret"`, `"token"` | Same field-preserving redaction | |
| Query strings: `password=value`, `token=value` | `password=<redacted>` | Same. |
| `Set-Cookie: session=value; Path=/; HttpOnly` | `Set-Cookie: session=<redacted>; Path=/; HttpOnly` | Cookie name preserved (proves cookie-based auth). Value redacted. |
| `Cookie: session=value` | `Cookie: session=<redacted>` | Same. |

### What redaction does NOT touch

- **Response status codes and HTTP shape** — the proof of "the exploit returned 200 OK" stays visible.
- **SQL injection payloads, XSS strings, and other exploit inputs** — these are the demonstration; redacting them would erase the finding.
- **Decoded JWT payloads** — `{"sub":"admin","role":"admin"}` stays visible. This is the confirmation evidence.
- **Response role / permission fields** — `{"user":{"role":"admin","permissions":["all"]}}` stays. Structural proof of what was bypassed.
- **`matched_patterns` and `timestamp` fields** — non-string-shaped metadata, no secret material.

### When redaction runs

- **Load-time only.** GuardLink reads the JSON file from disk, applies redaction once, and the redacted form is what every downstream consumer (dashboard, markdown report, SARIF, MCP queries from coding agents) sees.
- **The JSON files on disk are not modified.** Redaction is a read-time transform. Disabling the flag and re-running restores full evidence to the dashboard.
- **Idempotent.** Running redaction on already-redacted content is a no-op — the redaction markers themselves don't match any of the input patterns.

### Visual indicator

When redaction is active, the Pentest Findings tab in the dashboard displays a teal banner: *Evidence redaction: enabled — JWT signatures stripped, credential values masked. Claims and exploit payloads preserved.* This is so a reader of the dashboard can't be confused about whether they're seeing full or redacted data.

### Caveats

- **JWT payloads can still contain PII** — `{"email":"alice@example.com","name":"Alice"}` would survive redaction since email/name aren't credential field names. If your test data includes PII you need to scrub, that's a separate concern from credential redaction. Use test accounts with synthetic data, or post-process the JSON before sharing.
- **Novel token formats** — the redaction patterns target well-known shapes (JWT, common credential field names). A custom token format your application uses (`MyOrg-Token: xyz123`) won't be caught. If you need broader coverage, file an issue with the pattern and we'll add it.
- **Performance** — for typical scan results (handfuls of findings), the redaction pass is sub-millisecond. Large scans with hundreds of findings each containing many KB of response body may take meaningfully longer to load; measure if it becomes a concern.
- **Toggling at runtime** — if you enable redaction, view the dashboard, then disable, the dashboard does not auto-refresh. Re-run `guardlink dashboard` to regenerate.

## Quick reference

```bash
# Recommended baseline (do this once per project)
echo '.guardlink/pentest-findings/' >> .gitignore
echo '.guardlink/cxg-templates/' >> .gitignore

# Optional: enable surgical redaction (enterprise / compliance use)
guardlink config set redact-evidence true

# Check current state
guardlink config show

# Disable
guardlink config set redact-evidence false
```
